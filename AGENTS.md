# AGENTS.md - Working conventions for the Cogito Publish Themes gallery

This directory is **both** a collection of standalone themes for Cogito Publish
**and** a Cogito Publish site that renders them into a browsable showcase
(dogfooding). The source of truth is the current code plus this file.
Conventions here are specific to the gallery; the engine-level rules live in
the `cogito-publish` repo's own `AGENTS.md`.

## Language & style
- **Brand name is `Cogito Publish`; the CLI command is lowercase
  `cogito-publish`.** Write "Cogito Publish" in prose (e.g. "a Cogito Publish
  site"). Keep legacy `pyssg` identifiers only where the codebase still uses
  them technically: imports, `pyssg.config.py`, `.pyssg-cache`, and related code
  identifiers that have not been renamed yet.
- **Docs and code are English** (identifiers, comments, templates, CSS).
  Discussion/replies may use the maintainer's working language; committed files
  stay English. **No emoji.**
- Comments explain **why**, not **what**. Match the surrounding file's idiom,
  comment density, and naming.

## Repository layout
- **A folder is a theme iff it contains `layout.toml`** (the layout manifest).
  Themes live under **`themes/<slug>/`** - never at the repo root. Non-theme
  folders (`content/`, `layouts/`, `dist/`, `.pyssg-cache/`) are the showcase
  site's own infrastructure.
- A theme is a layout package:
  ```
  themes/<slug>/
    layout.toml        # manifest: name, version, default_template, [options]
    templates/         # Jinja2 (*.html.j2)
    assets/            # css/js/fonts -> copied to /assets/ in the demo output
    i18n/              # UI strings (en.toml, ...); a site may override them
    example/           # a minimal demo site (also the screenshot source)
    screenshots/       # committed PNG screenshots (source assets, NOT build output)
    STYLE.md           # provenance when CSS is compiled offline from upstream SCSS
    THEME-LICENSE      # upstream license when the theme is adapted from another
  ```
- **The gallery is itself a Cogito Publish site:** `pyssg.config.py` (site config),
  `content/showcase/<slug>.md` (one page per theme), `layouts/gallery/` (the
  gallery's own layout). It builds the [themes.gohugo.io]-style showcase.
- **A theme's `example/` resolves its layout via
  `Path(__file__).resolve().parent.parent`** - it tracks the theme it lives in
  and must keep working after the theme directory is moved. Do not hardcode
  theme paths in example configs.

## Running locally
- `cogito-publish` is provided by the sibling `../cogito-publish` project, so
  commands here use
  `uv run --project ../cogito-publish cogito-publish ...`. Inside the real monorepo (where this
  dir is `themes/`) the `--project` flag is dropped.
- Gallery: `uv run --project ../cogito-publish cogito-publish --site . serve` (default
  `127.0.0.1:8000`). One theme demo: `... cogito-publish --site themes/<slug>/example serve`.
- **`cogito-publish serve`'s incremental cache may not invalidate on
  template/CSS-only
  edits.** After editing `layouts/gallery/`, restart the dev server (or
  `rm -rf dist .pyssg-cache` and rebuild) to see the change.

## Registering a theme in the showcase
Three steps, all required:
1. Add the slug to `_THEMES` in `pyssg.config.py` so its screenshots mount:
   `themes/<slug>/screenshots` -> `/screenshots/<slug>` (mount sources resolve
   against the **site directory**).
2. Add `content/showcase/<slug>.md` (see the description format below).
3. Commit the theme's screenshots under `themes/<slug>/screenshots/`.

`.gitignore` keeps demo build artifacts out (`themes/*/example/dist/`,
`themes/*/example/.pyssg-cache/`); **screenshots are committed source assets.**

## Screenshots - exactly two fixed sizes
- **Capture every screenshot at one of exactly TWO fixed viewports - nothing
  else.** A third size makes the gallery layout hard to control.
  - **Desktop: 1280x800**
  - **Mobile: 390x844**
- All shots come from the theme's `example/` demo (`cogito-publish --site
  themes/<slug>/example build`, serve `dist/`, capture headless at the viewport
  above). Screenshots are committed, not generated by the Cogito Publish build.
- Keep the **same ordered set of views per theme** so the gallery is uniform;
  every file is unambiguously either a desktop or a mobile capture.

## Gallery - no layout shift from screenshots
- **The gallery, not the image's intrinsic size, controls screenshot
  dimensions.** Because captures are constrained to two known sizes, the gallery
  knows every aspect ratio ahead of time.
- **Reserve the slide box before the image loads** - render screenshots inside a
  fixed `aspect-ratio` container with `object-fit: contain` (letterbox mixed
  desktop/mobile shots against `--bg-soft`). Never let a loading or
  differently-sized image reflow the page (zero CLS). `loading="lazy"` is fine
  *because* the box is pre-sized.
- The showcase carousel (`layouts/gallery/templates/showcase.html.j2`) is the
  reference implementation: a scroll-snap track of uniform `aspect-ratio: 16/10`
  slides, prev/next arrows, and dot indicators kept in sync via
  `IntersectionObserver`. Keep it dependency-free and theme-aware (uses the
  gallery's CSS custom properties; respects `prefers-reduced-motion`).

## Theme description - one markdown per theme, fixed format
- **Every theme has exactly one markdown description, `content/showcase/<slug>.md`,
  rendered in the theme's detail page** (`/showcase/<slug>/`) by
  `layouts/gallery/templates/showcase.html.j2`. Follow this format:
  ```markdown
  ---
  title: <Display Name>
  excerpt: <one-sentence summary>            # shown as the lead and meta description
  tags: [<Category>]                          # first tag is the category chip
  upstream: <name or omit>                    # if adapted from another theme
  upstream_url: <url or omit>
  source_url: <link to the theme's source>
  features:                                   # bullet list rendered in the sidebar
    - <feature>
  images:                                     # screenshot URLs -> /screenshots/<slug>/...
    - /screenshots/<slug>/...
  ---
  <Markdown prose: what the theme is, who it is for, and how to adopt it
  (copy the theme dir, point Config.layout at it).>
  ```
- Keep the frontmatter keys and their meaning stable; templates read them by
  name. New optional keys are fine, but **do not rename or repurpose existing
  ones** without updating the template and this file.

## Changing a settled convention
Propose, get approval, then update this file in the same change. Keep
`README.md`, `pyssg.config.py`, `.gitignore`, and the example configs consistent
with whatever layout this file describes.
